# tele v1.0.0

The first stable release of `tele`, an unofficial Telegram MTProto CLI built for
bounded, auditable use by agents and humans.

## What shipped

- Safe explicit mutations with read-only and dry-run guards, deletion-scope
  routing, confirmed/rejected/unknown outcomes, and reconciliation handles.
- Truthful cursor-based read, search, and export pagination with explicit
  completeness, truncation, source-peer, sender, reply, thread, forward, edit,
  and album context.
- Stable `tele/v1` JSON envelopes, typed JSONL records, public JSON Schemas,
  strict allowlists, command goldens, and programmatic exit-code families.
- Exact message-body retrieval in JSON, terminal-safe human rendering, and an
  explicit untrusted-content/prompt-injection boundary.
- Atomic private local state, cross-process locking, permission repair, encrypted
  sessions backed by macOS Keychain, bounded deadlines/flood waits, and an
  aggregated read-only doctor.
- Current reviewed dependencies, pinned CI actions, race/static/security gates,
  deterministic macOS/Linux archives, checksums, provenance attestations, MIT
  licensing, private vulnerability reporting, and Homebrew distribution.

## Compatibility and migration

- Machine consumers from alpha.10 or earlier must accept `tele/v1` instead of
  `tele/v1alpha1` and resolve schemas under `schemas/v1`.
- Message bodies are returned exactly, including OTP- or credential-like text;
  the removed alpha redaction fields do not exist in v1.
- Literal login codes, 2FA passwords, and API hashes are not accepted through
  process arguments. Use hidden prompts or the documented environment inputs.
- Breaking command or machine-schema changes are reserved for a future major
  release. Minor releases may add commands without changing existing contracts.

## Platform scope

macOS arm64 and amd64 are supported with Keychain-backed secret storage. Linux
arm64/amd64 artifacts are preview builds until a native secret-store backend is
implemented. Windows amd64 remains compile-smoke only.
